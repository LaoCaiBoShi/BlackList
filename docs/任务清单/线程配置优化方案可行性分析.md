# 线程配置优化方案可行性分析

## 方案对比

| 方案      | 解压线程 | 消费者线程 | 总线程 | SSD   | HDD   |
| --------- | -------- | ---------- | ------ | ----- | ----- |
| **当前**  | 8        | 32         | 40     | 40>32 | 40>32 |
| **优化A** | 4        | 28         | 32     | ✅     | ❌     |
| **优化B** | 2        | 30         | 32     | ⚠️     | ✅     |

---

## 可行性分析

### 1. 队列缓冲能力

**关键问题**：解压线程减少后，队列能否持续供应JSON文件？

```
解压线程(4) → 队列 → 消费者线程(28)
     ↓                    ↑
   速度A              速度B（更快）
   
需要：速度A × 时间 ≥ 速度B × 时间 + 队列容量
```

| 场景 | 解压速度 | 解析速度 | 队列需要 | 可行性         |
| ---- | -------- | -------- | -------- | -------------- |
| SSD  | 快       | 快       | 中等缓冲 | ✅ 可行         |
| HDD  | 慢       | 快       | 大量缓冲 | ⚠️ 需要队列够大 |

### 2. 队列容量设计

```cpp
// 当前队列大小
config.queueSize = config.jsonThreadsPerProvince * provinceCount;

// 优化后：需要更大的队列缓冲
if (isSsd) {
    queueSize = cpuCount * 4;  // 128 (32核)
} else {
    queueSize = cpuCount * 8;  // 256 (HDD需要更大缓冲)
}
```

### 3. 风险评估

| 风险           | SSD场景      | HDD场景          | 应对措施         |
| -------------- | ------------ | ---------------- | ---------------- |
| 解压跟不上解析 | 低（解压快） | **高**（解压慢） | 增大队列缓冲     |
| 内存占用增加   | 低           | **高**（队列大） | 限制队列最大长度 |
| 消费者空闲     | 低           | **中**           | 监控并预警       |

## 推荐配置

### SSD配置（4 + 28）
```
解压线程: 4
消费者线程: 28 (填满CPU)
队列大小: 128
```

| 优点         | 缺点               |
| ------------ | ------------------ |
| 减少线程争抢 | 解压可能轻微跟不上 |
| CPU专注解析  | -                  |
| 内存占用适中 | -                  |

### HDD配置（2 + 30）
```
解压线程: 2
消费者线程: 30
队列大小: 256
```

| 优点           | 缺点               |
| -------------- | ------------------ |
| 减少IO争用     | 解压可能明显跟不上 |
| 顺序读写更高效 | 消费者可能部分空闲 |
| -              | 需要更大队列缓冲   |

---

## 可行性结论

### SSD场景 ✅ 可行
- 4个解压线程足以供应28个消费者
- 队列缓冲128个JSON文件
- 解压速度快，能持续供应

### HDD场景 ⚠️ 可行但需注意
- 需要更大的队列缓冲（256+）
- 监控消费者空闲情况
- 可能需要动态调整
